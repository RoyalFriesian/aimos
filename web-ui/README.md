# AimOS Web UI Architecture & Organization

This README serves as the primary technical documentation for the AimOS Web UI. It is designed to help AI agents and human developers quickly understand the structure, scale safely, and parse the application's flow.

## 1. Directory Structure

- 📁 `web-ui/src/`
  - 📄 `main.tsx`: Entry point. Injects React into the DOM and imports base CSS.
  - 📁 `app/`
    - 📄 `App.tsx`: Main routing and state container. Connects the API, themes, and views.
    - 📄`types.ts`: Core TypeScript definitions mappings for backend domains Models (`Project`, `Thread`, `Agent`, `Message`).
    - 📁 `api/`: Networking and client-side fetches (`client.ts`).
    - 📁 `components/`: UI Components.
      - 📁 `ui/`: Generic, reusable Tailwind + Radix-like primitive UI views.
      - 📄 `MindmapView.tsx`: Core visual map of threads/tasks.
      - 📄 `ChatPanel.tsx`: Conversation interface with the CEO agent.
      - 📄 `Sidebar.tsx`: Project and navigation tree.
    - 📁 `data/`: Mock implementations (e.g. `mockData.ts`).
  - 📁 `styles/`
    - 📄 `index.css`: Globals, Tailwind Directives, and custom Base layer definitions.
    - 📄 `theme.css`: Theme variables (using OKLCH colors and CSS custom properties) mapped to light/dark modes.
    - 📄 `fonts.css`: Font declarations.
    - 📄 `tailwind.css`: Utility classes generated by Tailwind.

## 2. JavaScript / TypeScript Features

### State Management
State is largely localized React state (`useState`, `useEffect`) grouped in `App.tsx` and distributed downwards via props or Context (`ReactFlowProvider`, `ThemeProvider`).
- **Scalability Note:** As the app grows, consider migrating deep prop threads to `zustand` or `React Context` to avoid prop drilling, specifically within `MindmapView` and `ChatPanel`.

### Data Models (`types.ts`)
The entire domain operates around `Thread` arrays representing hierarchical tasks and messages.
- AI Agents are linked to specific messages and threads.
- **Organization Request:** Any new domain models should exclusively go into `types.ts` and heavily employ JS/TS-Doc (`/** ... */`) commenting to declare exactly what fields represent.

### API Layer
- Currently mapped in `src/app/api/client.ts`. Contains implementations to fetch top-level `Project` contexts and dive into their `Thread` execution states.

## 3. Styling & Custom Properties (CSS)

The UI relies heavily on Tailwind v4 and CSS variables (`web-ui/src/styles/theme.css`).

### Design System (OKLCH)
- Theme colors use the OKLCH color space (`oklch(lightness chroma hue)`), which enables perceptual uniform brightness across dark and light modes.
- The default colors are built into CSS custom properties (`--primary`, `--background`, `--sidebar`, etc.).

### Scaling Styles
- **Do not** scatter colors and hardcoded hex values in `.tsx` files.
- **Always** use predefined Tailwind variables which reference `theme.css` via the `bg-background`, `text-primary` paradigms. This preserves the dark/light mode toggle (`ThemeProvider.tsx`).
- Layers (`@layer base`) in `index.css` define the fallback styles for default HTML elements to keep HTML typography consistent before Tailwind utility override.

## 4. Agent Reading Guide

**Dear AI Agent Context Contextualizer:**
When analyzing this application:
1. Start at `app/types.ts` to understand the domain data.
2. Read `app/App.tsx` perfectly understand where API data merges into Client State.
3. Understand styling stems from `styles/theme.css` + Tailwind standard classes.
4. Modify views inside `components/` while always ensuring type-safety. Never break `types.ts` interfaces without also reflecting changes in `api/client.ts`.
